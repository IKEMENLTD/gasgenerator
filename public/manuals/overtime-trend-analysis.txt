========================================
残業時間トレンド分析 - 使い方説明書
========================================

----------------------------------------
■ このシステムについて
----------------------------------------

「残業時間トレンド分析」は、企業の残業時間を可視化・分析し、
従業員の健康管理と労働環境の改善を支援するウェブアプリケーションです。

こんなことに使えます:
- 従業員の日々の残業時間を記録・管理する
- 部門ごとの残業時間の推移をグラフで把握する
- 残業が多い従業員や部門を早期に発見し、アラートで通知する
- 面談や業務調整などの介入記録を残して追跡する
- 週次・月次・年間の残業時間レポートを自動生成する
- CSVエクスポートで外部ツールと連携する

----------------------------------------
■ 画面構成と機能説明
----------------------------------------

【1. ダッシュボード】

アプリを開くと最初に表示される画面です。

- サマリーカード
  有効ユーザー数、部門数、未対応アラート数、
  平均残業時間の4つの指標が一目で確認できます。

- 部門別残業時間トレンドグラフ
  過去12週間の残業時間推移が折れ線グラフで表示されます。
  赤い点線は目標上限を示しています。

- 最近のアラート一覧
  直近のアラートが表形式で表示されます。
  未対応のアラートは赤色、対応中は黄色で強調されます。

- 部門サマリー
  直近週の部門ごとの合計残業時間、平均残業時間、
  超過人数が確認できます。

----------------------------------------

【2. 残業時間登録】

従業員の残業時間を手動で登録する画面です。

入力項目:
- ユーザー選択: 氏名で検索して選択します
- 記録日: カレンダーから選択（未来の日付は不可）
- 残業開始時刻: 例) 18:00
- 残業終了時刻: 例) 20:30
- 休憩時間(分): 例) 15
- 申請理由: 任意入力

入力すると「実残業時間」が自動計算されて表示されます。
「登録する」ボタンを押すとスプレッドシートに保存されます。

----------------------------------------

【3. 残業時間一覧】

登録された残業時間記録を一覧で確認する画面です。

- フィルター機能
  ユーザー、期間（開始日〜終了日）、
  申請ステータスで絞り込めます。

- ソート機能
  テーブルの列ヘッダーをクリックすると
  昇順/降順でソートできます。

- 承認/却下操作
  「申請中」のレコードに対して承認・却下ができます。

- ページネーション
  20件ずつ表示され、ページを切り替えて閲覧できます。

ステータスの種類:
  未申請 / 申請中 / 承認済 / 却下 / 差し戻し

----------------------------------------

【4. ユーザー管理】

従業員の情報を登録・編集する画面です。

- ユーザー一覧テーブル
  登録済みユーザーの一覧が表示されます。
  検索バーで氏名を検索できます。

- ユーザー追加
  「ユーザー追加」ボタンからダイアログを開き、
  以下の情報を入力して登録します。
  氏名、ふりがな、メールアドレス、所属部門、
  役職、入社年月日、雇用形態、管理者権限、ステータス

- ユーザー編集
  一覧の「編集」ボタンから既存ユーザーの情報を修正できます。

----------------------------------------

【5. 部門管理】

組織の部門情報を登録・編集する画面です。

- 部門一覧テーブル
  部門名、親部門、部門長、拠点名、ステータスが表示されます。

- 部門追加/編集
  部門名、親部門（階層構造対応）、部門長、
  設立年月日、拠点名、ステータスを設定できます。

----------------------------------------

【6. アラート管理】

残業時間超過アラートを管理する画面です。

- フィルター
  対象種類（ユーザー/部門）、アラート種類、
  解決ステータス、期間で絞り込めます。

- アラート一覧
  未対応は赤色背景、対応中は黄色背景で強調表示されます。

アラートの種類:
  個人週次超過 / 部門平均超過 / 法定時間超過

解決ステータス:
  未対応 / 対応中 / 解決済 / 放置

----------------------------------------

【7. 介入記録】

アラート発生後の対応内容を記録・管理する画面です。

左側: 介入記録一覧
  過去の介入記録が時系列で表示されます。

右側: 介入記録登録フォーム
  - 対象アラート: アラートIDで検索して選択
  - 介入日時: 日時を入力
  - 介入方法: 面談/メール/業務調整/健康指導から選択
  - 介入内容: 具体的な内容を記入（必須）
  - 次のアクション: 今後の予定を記入
  - ステータス: 完了/継続中/中断から選択

登録すると、関連するアラートのステータスも自動更新されます。

----------------------------------------

【8. レポート】

各種レポートを生成・閲覧する画面です。

レポートの種類:
- 個人週次サマリー: 特定ユーザーの週次残業時間推移
- 部門月次サマリー: 特定部門の月次残業時間集計
- 全社トレンド分析: 全社的な残業時間の傾向分析

操作手順:
1. レポート種類を選択
2. 対象ユーザーまたは部門を選択
3. 期間（開始日〜終了日）を指定
4. 「生成」ボタンをクリック

結果はグラフとテーブルで表示されます。
「CSVエクスポート」ボタンでデータをダウンロードできます。

----------------------------------------

【9. システム設定】

システムの各種設定を管理する画面です。
4つのタブで構成されています。

[アラート閾値設定]
  個人週次残業上限、部門平均週次残業上限、
  法定残業時間上限などの閾値を変更できます。
  各項目の数値を変更して「保存」ボタンを押します。

[通知テンプレート管理]
  アラート通知メールの件名と本文を編集できます。
  テンプレート変数（${USER_NAME}等）が使用可能です。

[システム設定値]
  システム名、データ保持期間、メール通知の有効/無効、
  年度開始月などの基本設定を変更できます。

[テストデータ]
  システムの動作確認用のサンプルデータを管理します。
  - テストデータ挿入: 部門5件、ユーザー12件、
    残業記録約700件などのサンプルデータを一括生成
  - 全データクリア: 全シートのデータを一括削除
  - シート状態確認: 各シートのデータ件数を確認

----------------------------------------

【10. 操作ログ】

システム上の全操作が記録されている画面です。

- フィルター
  期間、ユーザー、カテゴリで絞り込めます。

- ログ一覧
  操作日時、ユーザー、カテゴリ、操作内容、
  詳細が確認できます。

----------------------------------------
■ モバイル操作について
----------------------------------------

768px以下の画面幅ではモバイルレイアウトに切り替わります。

- 画面左上のハンバーガーメニュー（三本線アイコン）を
  タップするとナビゲーションメニューが開きます。
- メニュー外をタップすると閉じます。
- テーブルは横スクロールで閲覧できます。
- グラフは画面幅に合わせて自動リサイズされます。

----------------------------------------
■ データについて
----------------------------------------

全てのデータはGoogleスプレッドシートに保存されます。

スプレッドシートは18個のシートで構成されています:
  ユーザーマスタ / 部門マスタ / 役職マスタ /
  規定値マスタ / 通知テンプレートマスタ /
  週次目標マスタ / 日次残業時間記録 /
  外部システム連携ログ / アラート履歴 / 介入記録 /
  コメント・フィードバック / 週次残業時間集計 /
  月次残業時間集計 / 部門別週次サマリー /
  年間残業時間サマリー / トレンド分析データ /
  システム設定 / 操作ログ

----------------------------------------
■ 注意事項
----------------------------------------

- 初回アクセス時はGoogleの承認画面が表示されます。
  「承認」をクリックしてください。
- データの登録・更新時は処理中のローディング画面が
  表示されます。完了まで操作をお待ちください。
- 大量のデータを扱う場合、処理に時間がかかる
  場合があります。
- 重要なデータを削除する前にスプレッドシートの
  バックアップを取ることをお勧めします。
